`promo_codes` is a single table with four flavors of code, distinguished by `type` (`'referral' | 'ambassador' | 'gift' | 'promo'`).

- **`referral`** — a personal code belonging to a regular user; earns them + the redeemer some coins.
- **`ambassador`** — same shape but higher rewards + monthly cap for approved ambassadors.
- **`gift`** — grants a subscription plan (via `gift_plan`, `gift_duration`). Typically single-use.
- **`promo`** — general marketing / internal-use.

Redemptions land in `code_redemptions`, one row per (code, redeemer) pair.

## Real column names

`promo_codes`:
- `id`, `code` (unique text), `type` (union above)
- `owner_id` (nullable) — the user who owns/earns from this code
- `coins_referrer`, `coins_redeemer` — reward amounts (integers)
- `gift_plan`, `gift_duration` — populated for `gift` type
- `max_uses`, `current_uses` — single-use gates via `current_uses < max_uses`
- `requires_action` — default `'first_upload'`; controls when the redeemer's reward flips from pending to granted
- `recipient_email` — for targeted gift codes
- `expires_at`, `is_active`, `created_at`

`code_redemptions`:
- `id`, `code_id`, `redeemer_id`
- `referrer_rewarded`, `redeemer_rewarded` — booleans; flip to true after the qualifying action fires
- `qualifying_action_at` — when the redeemer completed the qualifying action
- `device_fingerprint`, `ip_address` — anti-abuse
- `created_at`
- UNIQUE (`code_id`, `redeemer_id`) — one redemption per person per code

There is **no** `redeemed_by` / `promo_code_id` / `redeemed_at` / `applied_plan` — those were fabrications. Cross-check `src/db/schema.ts` before editing.

## Ambassador program

An ambassador applies (currently manual — no `applications` table yet). Once approved, CTO manually INSERTs a row:

```sql
INSERT INTO promo_codes (code, type, owner_id, coins_referrer, coins_redeemer, requires_action, is_active)
VALUES ('RYAN10', 'ambassador', '<user-uuid>', 15, 10, 'first_upload', true);
```

The ambassador shares their code. When someone redeems (during onboarding step 2 or later from `/redeem-code`):

- `POST /api/v1/codes/redeem { code }` looks up the row.
- Applies a benefit to the redeemer (usually 10 bonus coins) and marks their reward pending.
- Inserts into `code_redemptions`.
- Owner's reward stays pending until `qualifying_action_at` fires (after their first wardrobe upload).

**Monthly cap** for the ambassador owner is enforced at redemption time via SQL — a running count of the owner's redemptions this month vs a per-tier cap.

## Gift codes

Generated by the CTO via SQL. Typically 8-char alphanumeric. Single-use:

- `POST /api/v1/codes/redeem` — checks `type='gift'`, `current_uses < max_uses`.
- Activates the associated `gift_plan` for `gift_duration`.
- Increments `current_uses`.

## Pending code from deep links

If someone clicks an ambassador link on a browser without the app installed, the landing site's `/invite/[code]` page:

1. Calls `GET /api/v1/codes/info/:code` to preview.
2. Sends them to App Store / Play (with the code embedded as a universal link parameter on iOS / Play Install Referrer on Android).
3. On first app open after install, mobile checks the parameter, writes it to AsyncStorage as `pendingCode`.
4. Onboarding step 2 (referral-code) reads it, pre-fills the field, clears the AsyncStorage key.

If the user backs out mid-onboarding, we clear `pendingCode` to prevent an infinite loop.

## Code validation

`POST /api/v1/codes/redeem`:

1. Look up by exact match (case-sensitive).
2. If not found → 404 `INVALID_CODE`.
3. If `is_active=false` or `expires_at` past → 400.
4. If `current_uses >= max_uses` → 409 `ALREADY_REDEEMED`.
5. If owner is the redeemer → 403 `CANT_REDEEM_OWN_CODE`.
6. If ambassador monthly cap hit → 429 `AMBASSADOR_LIMIT_REACHED`.
7. Otherwise apply + insert + increment.

The response mirrors the applied effect.

## Trust model

- Ambassador codes are memorable strings (`RYAN10`, `CLOSETGO`). Not secrets.
- Gift codes are unguessable random strings. Distributed via marketing.
- Redemption always requires being signed in — no anonymous redemptions.
